# AGENTS.md — math 项目协作契约

## 项目目的

本项目不是通用数学学习辅助工具。它的核心目标是：**基于一个使用者的日常练习、错题、错因与复习记录，形成个人薄弱图谱，并给出针对性的强化复习训练建议，从而提升复习效率**。

通用知识点、通用题型、通用技巧和通用易错点只是基础底座；真正的产品价值在于识别“这个人现在最该补什么、为什么、下一步练什么”。

## 北极星

所有功能设计、数据建模、界面文案和推荐逻辑都优先回答：

1. 我当前真正薄弱的节点是什么？
2. 这些薄弱点来自哪些错题、错因或练习证据？
3. 下一组训练题为什么能更高效地补这些薄弱点？
4. 练完以后，个人薄弱图谱如何更新？

如果一个功能不能帮助以上问题，应降低优先级。

## 图谱设计原则

- 不要把知识库建成单棵章节树。章节树只是浏览入口。
- 系统应维护两层图：
  - **基础学习图谱**：知识点、题型、技巧、通用易错点、题目之间的关系。
  - **个人薄弱图谱**：基于 attempts、mistakes、mistake_diagnoses 形成的个人薄弱点、掌握度和证据链。
- 区分通用易错点和个人易错点：
  - `common_pitfalls` 是通用资料里的常见错误。
  - `personal_weaknesses` 是当前使用者通过真实练习暴露出来的薄弱节点。
- 推荐、学情和训练入口应以 `personal_weaknesses` 为主，而不是只按章节错误率或通用题型排序。

## 实现规则

- 优先保持单设备、本地优先、可解释、可备份。
- 前端继续保持 vanilla JS + ESM + MathJax，不引入构建工具，除非后续明确改方案。
- 后端优先 FastAPI + SQLModel + SQLite。
- 不新增云依赖；OCR/LLM 能力必须有本地降级路径。
- 不要为了抽象而抽象。先用清楚的领域表表达业务，再在需要跨实体推理时使用图谱边。
- 题目入库时，尽量保留 Top-K 候选关系，允许人工校正，不要过早把题目压成单一知识点。
- 错题不是终点。每次错误都应尽量沉淀为错因诊断，并更新个人薄弱图谱。

## 数据建模规则

- `knowledge_points` 继续兼容现有 markdown 解析和 `slugify` ID，避免破坏已有链接。
- `question_patterns`、`skills`、`common_pitfalls` 表达基础学习图谱。
- `mistake_diagnoses` 表达“这次为什么错”。
- `personal_weaknesses` 是学情和推荐的核心表。
- 推荐算法不得只使用 `kp` 错误率；必须综合知识点、题型、技巧、易错点和个人证据。
- 所有自动诊断和 LLM 诊断都应保留 `source` / `confidence` / `evidence`，并允许人工修正。

## 用户体验规则

- 学情页面优先展示个人薄弱 Top N、证据错题、建议训练，而不是泛泛统计。
- 推荐题必须可解释：说明命中的个人薄弱点、关联题型/技巧/易错点、推荐原因。
- 错题详情页应支持“我错在...”的快速标注和复盘笔记。
- 不要把“学习效率提升”稀释成打卡、排行榜、泛内容浏览等次要功能。

## 验证规则

- M0/M1 阶段要持续使用 POC 脚本验证：
  - 知识点 markdown 是否稳定结构化。
  - 讲义/docx/题目样本是否能抽出题型和典例。
  - 题目到知识图谱的 Top-K 关联是否可审、可修正。
- DOCX 讲义抽取 POC 使用 OOXML 文本节点（`word/document.xml` 中的 `w:t` / `m:t`）和题型标记规则，后续应沉淀为可复用导入服务，不要当作一次性脚本丢弃。
- 新增推荐或学情逻辑时，必须用可读样例解释“为什么推荐这道题”。
- 完成里程碑时，更新 `PLAN.md` 的实际结果、风险和验证证据。

## 文档维护

- 架构和产品决策写入 `PLAN.md`。
- 项目级长期约束写入本 `AGENTS.md`。
- POC 产物保留在 `poc/`，用于验证图谱建模是否符合真实材料。
