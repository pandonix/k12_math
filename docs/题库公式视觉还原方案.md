# 题库公式视觉还原方案

## 一、问题

题库里来自 Word 文档的题目，公式显示和原始题目完全不一样：希腊字母变乱码、分数被压成一行、上下标错位。典型例子（某概率题解析）：

| 原始 | 旧版显示 |
|---|---|
| `Ω₁ = {(B₁,B₂),(B₁,G₁),…}` | `Ù,1=,{(B1,B2),(B1,G1),…}` |
| `P(A) = 8/12 = 2/3` | `P(A)=1⁸2=²3` |
| `P(A) = 8/16 = 1/2` | `P(A)=1⁸6=¹2` |

## 二、根因

题库来源是 `data/uploads/` 下两份 Word 文档。分析后发现两份文档的数学公式**存储方式完全不同**：

| | 文档1：62个方法技巧 | 文档2：26个易混易错 |
|---|---|---|
| OMML 结构化公式 | **1818 个**（含 1199 分数 / 1322 根号 / 532 上标） | 仅 6 个 |
| MathType OLE 公式 | 250 个 | **2293 个**（几乎全部） |
| 渲染兜底图 `.wmf` | 201 | 1645（真实矢量图） |

导入脚本对两类公式都处理得不对：

1. **OMML 公式**：只抓取 `<m:t>` 纯文本拼接，丢掉了分数 / 根号 / 上下标的结构。例如 `¼a` 变成 `14a`，选项 `-1/4` 变成 `14`——不只是难看，**语义都错了**。
2. **MathType OLE 公式**：旧的 `wmf_to_svg()` 从 WMF 字节流里"抠文本"再硬拼成一行，用 `cp1252` 误解码（`Ω`→`Ù`），并把按坐标定位的字符压平——这就是截图里乱码的来源。

关键前提：前端 `index.html` 早已加载 MathJax 3（tex-svg），`app.js` 的 `renderMarkdown / formatInline / typeset()` 也已支持 `$…$`、`$$…$$` 和动态排版。**渲染层一直就绪，问题纯粹在导入层从未产出 LaTeX。**

## 三、解决方案（按公式类型分两条链路）

### 链路 A：OMML → LaTeX（覆盖 1824 个，文档1 主力）

新增 `scripts/omml2latex.py`——纯标准库的递归转换器，把每个 `m:oMath` 整体转成 LaTeX，输出 `$…$`，交给已就绪的 MathJax 渲染。

- 覆盖：文本 / 分数 `m:f` / 上下标 `sSup·sSub·sSubSup` / 根号 `rad` / 括号 `d` / n 元运算符 `nary` / 函数 `func` / 重音 `acc` / 上下划线 `bar` / 群字符 `groupChr` / 上下极限 `limLow·limUpp` / 矩阵 `m`。
- Unicode 运算符与希腊字母映射到 LaTeX 命令；全角标点 `，、；：` 归一化为半角。
- 优势：可选中、可搜索、跟随主题色、能换行重排。

实测结果（文档1）：

```
$\frac{b^2+c^2-a^2}{2bc}$          # 余弦定理
$\frac{\sqrt{2}+\sqrt{6}}{4}$
$\left(\frac{\pi}{3},\frac{4\pi}{3}\right)$   # 区间
$\frac{1}{2}\sin A+\frac{\sqrt{3}}{2}\cos A=1$
```

### 链路 B1：MathType WMF → 真实渲染图（覆盖 2543 个，文档2 主力）

彻底删掉旧的 `wmf_to_svg()` 抠文本逻辑，改用真正的渲染器（新增 `scripts/wmf_render.py`）：

1. 用 **LibreOffice headless** 把 Word 真实显示用的 `.wmf` 矢量图渲染成 A4 @ 300 DPI 的 PNG（逐像素忠实原文档）。
2. 用 **Pillow** 自动裁掉 A4 页面空白，并把白色背景抠成透明（保留字形抗锯齿）。
3. 显示尺寸取自文档里 `<v:shape style="width:..pt;height:..pt">` 的**声明尺寸**（pt × 96/72 → CSS px），烘焙进图片 token；缺失时回退到裁切像素折算。
   - 关键：Word 是按 shape 的 pt 尺寸缩放 OLE 对象的，**不是** WMF 内禀尺寸。单符号公式（如 `B₁`）的 WMF 内禀坐标空间往往很大、被 Word 缩小数倍——若按内禀尺寸折算会严重偏大。实测覆盖率 doc1 201/201、doc2 1644/1645。

> 选择 B1（图片）而非 B2（解析 MathType MTEF 二进制转 LaTeX）：MTEF 格式难啃、无可靠现成库、2293 个成功率不确定。**采用先 B1 后 B2 的分阶段策略**——阶段一先用 B1 让全部公式视觉还原；阶段二再逐步把图片升级为 LaTeX（换取可选中/可搜索）。

### 链路共用：尺寸与前端

- 图片 token 扩展为 `[[img:path|宽x高]]`，宽高为真实印刷尺寸的 CSS 像素。
- `app.js` 的 `formatInline` 解析尺寸并写成行内 `style="width:..;height:.."`，公式图加 `.docx-equation` 类（去掉旧的 `max-height:120px` 限制——否则 300 DPI 大图会被一律撑到 120px）。
- 底层 PNG 是高 DPI 大图，按印刷尺寸缩小显示，因此在 Retina 屏上依然清晰。
- 前端 MathJax 动态排版（`typeset()`）此前已在三个阅读视图接好，无需改动。

## 四、改动清单

| 文件 | 改动 |
|---|---|
| `scripts/omml2latex.py` | **新增**：OMML → LaTeX 递归转换器 |
| `scripts/wmf_render.py` | **新增**：WMF → 裁切透明 PNG（LibreOffice + Pillow），返回印刷尺寸 |
| `scripts/import_uploaded_docx_questions.py` | `extract_docx` 改为把 `m:oMath` 转 LaTeX；`export_all_images` 改用批量 WMF 渲染（带按文档缓存 + 已存在即跳过的快速路径）；`parse_shape_display_sizes` 读取 v:shape 声明尺寸；`image_token` 输出 `[[img:path\|宽x高]]` |
| `renderQuestionImages`（app.js） | 附图区去重兼容带尺寸后缀的 token，避免内联公式图被放大重绘 |
| `app.js` | `formatInline` 解析 `\|宽x高` 并应用行内尺寸 |
| `styles.css` | 新增 `.docx-equation` 规则 |

## 五、依赖

- **LibreOffice**（headless）：`brew install --cask libreoffice`。`wmf_render.py` 会在 `/Applications/LibreOffice.app` 及 `PATH` 中自动查找 `soffice`。
- **Pillow**：`pip install Pillow`（已装入项目 `.venv`）。

## 六、重新导入

```bash
source .venv/bin/activate
python3 scripts/import_uploaded_docx_questions.py     # 自动备份 DB 到 data/*.bak
```

首次会渲染全部 WMF（约几分钟）；之后因有"已存在即跳过"快速路径，几秒即可完成。

## 七、验证

截图那道题（id=412）重新导入后：`Ω₁` 正确、集合花括号与下标 `B₁B₂G₁G₂` 完整、`P(A)=8/12=2/3`、`P(A)=8/16=1/2` 均为带分数线的真分数，与原文档一致。
